- Question 39 decision: choose E. Starting identity visibility shows name, short description, possible routes, and approximate risk, while hidden secrets, true risk, special NPCs, and family details are generated backstage by AI.

- Runtime JSON IDs, tags, progress IDs, route tags, ending tags, and other searchable values should use snake_case stable IDs.
- AI output is a proposal only. The future engine must validate AI JSON and state patches before applying changes to an authoritative save.
- Normal playable events default to 3 rich choices, but death, ending, clarification, forced consequence, pure display, and system-only transition responses may use 0 choices or another interaction mode.
- AI response types include `life_event`, `action_resolution`, `clarification_request`, `forced_consequence`, `ending_summary`, `memory_update`, and `json_repair`.
- Hidden information can guide foreshadowing but must not be leaked in player-facing text before the save reveals it.
- Important or risky random checks should be generated by the engine; AI narrates the provided result rather than freely inventing success or failure.
- Player-visible changes should be structured separately from the authoritative `statePatch`.
- The first program skeleton should use mock AI before real provider integration so the engine contract can stabilize first.
- The next implementation layer should add save/load shape, real state patch application, deterministic rarity drawing, initial important NPC generation, and provider adapters.
- Accepted AI responses can now advance age, update world progression, append event history, add memory entries, and save the resulting run as JSON.
- Initial important NPCs are now system-generated from world templates, keeping player setup limited while giving each run persistent relationship anchors.
- The first playable MVP surface is CLI-based: players can choose 1/2/3 for AI-generated options, choose 4 for optional free-form attempted action, and save the resulting run.
- AI event `choices` arrays are limited to `choice_1`, `choice_2`, and `choice_3`; UI/CLI renders the optional free-form entry separately as item 4.
- The CLI playtest now has a player-facing setup wizard: choose world, enter limited identity fields, allocate 20 points, keep 3 talents from a draw of 5, then begin the simulated life.
- MVP short-run completion is now available: after the configured ending age, the play session emits an `ending_summary` with 0 choices and stores final ending state in the save.
- Real AI generation is opt-in through `--ai deepseek`; default `npm run play` remains deterministic mock AI for offline testing.
- Event pools are open soft seed pools, not fixed scripts or event whitelists.
- Normal event generation uses seven source types. Each world has different weights so cultivation, Cthulhu life, and wasteland runs feel distinct while staying inside the same life-simulator engine.
- Only `seed_pool` source events require selected seeds. Other source types can generate from current save context, world rules, NPCs, relationships, and history.
- Real AI mode should cover both `life_event` and `action_resolution`; otherwise the player can see AI events but still gets mock-only consequences after choosing or using free input.
- Real AI mode should also cover `ending_summary`; otherwise the final biography and score would not meet the playtest requirement for AI-generated life conclusions.
- AI patches must stay targeted. The engine applies allowed patch arrays but still does not let AI replace the whole save object.
- Talent drawing rolls rarity first, then selects a concrete not-yet-drawn talent from that rarity pool. If the target rarity is exhausted, MVP falls back to any remaining not-yet-drawn talent.
- Save files remain readable JSON, but they are no longer accepted blindly. A loaded run must match `mvp.run.v1` core shape before play continues.
- CLI status summaries expose backend systems to the player: potential/manifested/exposure values, world progress, statuses, rough NPC relationship ranges, and faction summaries.
- DeepSeek is now one configured real-AI provider, not the only supported real-AI path. Generic OpenAI-compatible mode requires explicit API key, base URL, and model to avoid guessing a paid provider or wrong model.
- Player-created identity remains limited. Players may provide name, gender, and a broad personality direction; detailed background, family, hidden secrets, and starting context remain system/AI generated from world, family background, attributes, talents, and personality direction.
- Clarification requests must not apply state changes before player confirmation. They explain risk and intent, then wait for `y`/`n`.
- Faction and location pools are generation hooks and constraints, not fixed scripts. They support joining, creating, exploration, conflicts, and NPC/context consistency.
- The current runtime content meets the fuller playtest minimum content target across all three MVP worlds. Future expansion should aim for ideal counts and better localization/prose coverage rather than only minimum counts.
- `check:playtest` treats real AI environment variables as advisory in local development: missing keys should warn rather than fail, because mock mode remains the offline development path.
- Missing DeepSeek/OpenAI-compatible API keys do not block local file work, mock-mode development, tests, validation, content audits, or provider request-shape tests; they only block live provider smoke tests.
