## Decisions

- Formal life timeline should contain lived/resolved history only. The current playable branch belongs in the bottom current node until the player acts.
- Action resolution is part of the life history. It should not be repeated as a separate full ordinary-player card.
- Early-years UI can derive stage nodes from the opening response, but it must not invent filler just to create a one-year-per-entry diary.
- Latest pacing decision supersedes the older stage-node wording: early-life UI must be strict age order from 0ň▓ü onward, while still avoiding empty yearly filler.
- Raw stable IDs remain allowed in backend JSON, schemas, logs, and GM/debug surfaces only.
- Fate preview must not reveal future triggers or route hooks. It may show ambiguous signs, rumors, strange objects, NPC reactions, or family concealment, while true hooks stay in `opening.hiddenHooks`, `opening.unresolvedThreads`, internal notes, or GM/debug surfaces.
- The formal web life journal starts at age 0 after `ň╝Çňžőń║║šöč`; early-life progression is part of the main timeline, not a separate ordinary-player page.
- Ordinary UI must show only the player character's current knowledge of NPCs. True identities, hidden roles, raw IDs, and future NPC functions stay backend/GM-only until story actions and state patches justify discovery.
- Talent details must be specific and world-aware, using stored effects/tags/rarity/manifestation data rather than generic filler text.
- Latest user changes override older project notes. When a later requirement changes the fate/opening flow, update docs and code immediately rather than preserving old wording for backward compatibility.
- Ordinary fate preview should never expose hidden NPC functions or unresolved-detail sections. Discovery belongs to later player action and justified `importantNPCUpdates`, while GM/debug may keep the full backend truth.
- Formal life page is the complete life journal and must start from age 0. The fate preview is only a confirmation page, not the early-life narrative surface.
- Ordinary player UI is not allowed to use placeholder labels for missing data. Missing safe Chinese names must be fixed, hidden, or repaired before rendering.
- Current playable events must show their age consistently with `timeSpan.ageEnd`, and the branch must follow from prior story context before presenting choices.
- Choice metadata is only fuzzy chance/risk. It must never pre-announce that a choice already succeeded or that the player already obtained, hid, extracted, defeated, or solved something.
