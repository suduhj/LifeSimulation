## Decisions

- Current saves treat `run.eventLog` as the replay authority when present. The run snapshot remains useful for compatibility and debugging, but load should rebuild from the event log first.
- DomainEvents are engine-owned. AI, mock, UI, and GM tools may propose changes, but reducers are the only code path allowed to settle those changes into the run projection.
- PlayerView, PromptView, and GMView are separate projections for different audiences. Ordinary player data should migrate toward PlayerView; GM/debug may keep full run and hidden fields.
- The MVP keeps legacy `run` fields available for current web compatibility while establishing PlayerView as the ordinary-player projection boundary.
- Selector Graph is the browser-panel CQRS layer. Reducers produce the authoritative run projection; selectors derive UI panel data; the browser should render `panelViews` first and keep raw run reads as compatibility fallback or GM/debug only.
- `panelViews.attributes.groups` is the ordinary attribute-panel contract. The browser may keep flat `attributes` for compatibility, but grouped cards are the source for new player-facing UI.
- When a child PR is merged into another feature branch, `main` is not updated automatically. If `main` still lacks those commits, create a follow-up PR from the updated feature branch into `main`.
- Annual yearly direction is curriculum-first. A long-running thread may echo in the background, but it should not own the year unless the engine promotes it through the annual contract.
- New annual director state must enter `statePatch -> DomainEvents -> reducer -> replay`; adding fields only to the snapshot is not acceptable.
- Ordinary text is Scene Object-first. Backend annual contracts, topic ledgers, asset ledgers, curriculum slots, and old-thread IDs may guide rendering, but they are not player-facing vocabulary.
- Family Background is an origin constraint, not an age-sealed growth stat. It must be resolved into a compatible world-specific origin before opening prose is shown.
