## Decisions

- The leak guard gates ONLY real AI provider output (inside `requestValidatedAiResponse`), not the shared `validateAiResponse`. Mock/dev/GM/engine responses are author-controlled and may legitimately carry backend ids in GM/debug surfaces, so applying the guard there caused false positives. Why: the leak risk is the provider parroting backend context; trusted engine content does not need the same gate. How to apply: when adding new provider response types, run `detectPlayerTextLeaks` alongside `validateAiResponse` in the provider path.
- The English-leak rule matches a 3+ word space-separated English phrase, not any long latin run. Why: a single long token can be a legitimate English player name and a hyphenated token can be a product name (OpenAI-compatible); only multi-word English phrases reliably indicate leaked backend sentences. Real backend ids are caught separately by the snake_case rule.
- Degenerate AI bodies fall back to the age-aware mock generator rather than fabricated filler. Why: fabricated filler was engineered to pass validation and masked AI failure, surfacing generic/again leaky text; mock is age-aware and world-specific. How to apply: do not reintroduce body fabrication in the repair normalizer; let validation fail and rely on the `safeGenerate*` mock fallback.
- The leak guard raises mock-fallback frequency (a leaky-but-otherwise-good provider response is rejected), and a mid-run mock event is jarring because the mock template is world-generic and ignores established threads. Mitigation chosen: make the repair pass fix leaks (rewrite to Chinese) so leak-only failures stay on the provider path instead of degenerating to mock. Why not just loosen the guard: leaking backend strings to players is worse than an occasional generic event. How to apply: when adding new player-facing leak rules, also extend the repair instructions so the leak is repairable, not only rejectable.
